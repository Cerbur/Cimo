# Cimo Engineering Rules

> 本文件承载长期工程规范。代理入口硬约束见 [../AGENTS.md](../AGENTS.md)，执行流程见 [agent_process.md](agent_process.md)。

随着项目推进，本文件会持续补充技术选型、代码约定和架构边界。新增规范必须能说明它解决的真实问题、当前必要性和可验证收益。

## 提示词集中维护

所有 system prompt、工具引导词、后续多 Agent/多 Step 的提示词都统一放在专门的 `prompt` 包中维护；入口层、Agent Loop、Client adapter 不直接内联大段提示词，只引用 `prompt` 包暴露的常量、模板或工厂方法。

## CLI 优先入口

当前阶段采用类似 Codex / Claude Code 的 CLI 交互模式，`./gradlew bootRun` 启动后直接进入 `>` 提示符，用户输入自然语言；入口层使用 JLine REPL 接入 Spring 生命周期，不采用 Spring Shell 的命令模式，也不引入 `ask ...` 这类显式命令壳。

## Spring 注入规范

除构造器中存在真实初始化逻辑、派生对象创建、不可变状态计算或启动期校验逻辑的类外，Spring Bean 依赖统一使用成员变量上的 `@Autowired` 显式注入；不要把仅用于字段赋值的依赖塞进构造器。

例外示例：`/agent/DefaultAgentLoop.java:31-34` 中通过 `ClientFactory.createClient()` 创建并保存 `Client`，构造器承担了初始化逻辑，可以保留构造器注入。

调整注入方式时需要同步检查测试构造方式和配置绑定方式。

## 中文注释约束

新增或修改代码时，类需要补充清晰的中文注释，说明其职责、输入输出或关键约束。

函数/方法应先评估其重要性、复杂度和复用范围，重要的业务方法、公共方法、流程编排方法或带有关键约束的方法需要补充中文注释，简单私有辅助方法、样板方法、显而易见的 getter/setter 等不重要方法无需为了形式补充注释。

核心业务逻辑、关键分支、状态流转和不易一眼看懂的实现细节，需要增加必要的中文行注释，注释应解释“为什么这样做”和“关键意图”，避免重复代码字面含义。

## 注释质量要求

注释的目标是补充代码本身难以表达的业务意图、设计约束、状态流转、边界假设、兼容性原因或历史决策，帮助人和后续 LLM 代理减少误判。

禁止为了凑注释而复述代码字面含义，例如“获取名称”“设置字段”这类机械注释。能通过清晰命名和简单结构表达的内容，优先通过代码本身表达；只有当“为什么这样做”或“不能随意改动的约束”不明显时，才补充必要注释。
